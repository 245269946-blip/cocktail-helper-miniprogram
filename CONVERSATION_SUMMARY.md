# 调酒小程序历史对话汇总

本文件把前几轮关于“调酒助手微信小程序”的讨论、产品方向、页面结构、视觉方向、开发结果和后续注意事项统一沉淀到当前项目中。

当前项目目录已经导入最新版小程序源码；历史版本、预览脚本和截图材料放在 `_history/`。

## 1. 项目核心定位

产品定位：

鸡尾酒配方搜索 + 风味推荐 + 场景购买指南小程序。

它不是内容流，也不是鸡尾酒百科，而是一个“饮品决策助手”。用户来这里主要是解决这些问题：

- 我想调一杯，但不知道怎么调。
- 我手上有酒或饮料，但不知道能搭什么。
- 我现在去便利店或超市，想知道买什么就能做。
- 我不知道自己适合喝什么，希望系统帮我快速推荐。

核心判断：

先满足确定需求，再诱导模糊需求。

也就是说，用户可能只搜索“金汤力”“可乐能调什么”“便利店调酒”“不苦”“低酒精”，页面要先给一个明确答案，再顺手提供更甜、更清爽、更便利店、更低酒感、更适合新手的方案。

## 2. 首页产品方向

首页不能做成内容流，应当做成搜索型功能首页。

第一优先级是搜索框，搜索占据产品第一入口。搜索文案可以是：

搜鸡尾酒 / 基酒 / 材料 / 风味 / 便利店搭配

需要支持的典型搜索词：

- 鸡尾酒名：莫吉托、金汤力
- 基酒：伏特加、威士忌、金酒、百利甜
- 饮料/材料：雪碧、可乐、橙汁、咖啡、牛奶
- 风味：甜口、不苦、低酒精、清爽、奶香
- 场景：女生喝、便利店调酒、家里有什么、聚会

热搜词：

莫吉托 / 金汤力 / 百利甜 / 便利店调酒 / 甜口 / 新手入门 / 低酒精 / 聚会

首页曾讨论过 6 个快捷入口：

- 搜配方：承接明确搜索。
- 按基酒找：金酒、伏特加、威士忌、朗姆、龙舌兰、利口酒等。
- 按材料找：可乐、雪碧、气泡水、柠檬、橙汁、牛奶、咖啡、茶等。
- 便利店调酒：项目差异化重点。
- 家里有什么：用户勾选已有材料，系统推荐可做酒单。
- 新手推荐：服务不知道搜什么的用户。

后续版本把首页进一步调整成行动入口优先：

- 去便利店买
- 家里有什么
- 找一杯好喝的
- 第一次尝试

## 3. 页面结构汇总

### 搜配方页

用户搜索“莫吉托”等明确酒名后，结果不只是给配方，而是进入结果页和详情页。

详情需要包含：

- 基础配方
- 调制步骤
- 风味说明
- 新手难度
- 替代材料
- 便利店版本
- 相似推荐
- 适合场景

其中“便利店版本、相似推荐、适合场景”是区别于普通配方库的关键。

### 按基酒找页

入口问题：

你手上有什么酒？

基酒分类包括：

金酒 / 伏特加 / 威士忌 / 朗姆 / 龙舌兰 / 百利甜 / 野格 / 梅酒 / 果酒 / 起泡酒

基酒详情页不是百科介绍，而是回答：

你现在能用这瓶酒做什么？

需要展示：

- 适合什么风味
- 经典配方
- 新手友好配方
- 便利店能搭什么
- 甜口版
- 清爽版
- 更容易接受版
- 进阶版

### 按材料找页

这个入口承接长尾搜索，例如：

- 可乐能调什么
- 雪碧能调什么
- 橙汁能调什么
- 咖啡能调什么
- 牛奶能调什么

材料分类包括：

碳酸饮料 / 果汁 / 茶饮 / 咖啡 / 乳饮 / 水果 / 冰块糖浆小料

材料详情页需要展示：

- 适合搭什么酒
- 经典搭配
- 便利店版本
- 低酒精版本
- 甜口版本
- 不容易翻车版本
- 不推荐搭配提醒

### 便利店调酒页

这是主打差异化页面。

第一屏不要问复杂问题，先问：

你现在在哪？

可选场景：

便利店 / 超市 / 家里 / 宿舍或租房 / 聚会前临时采购

然后再问：

你想要什么感觉？

风味可选：

清爽 / 甜口 / 酸甜 / 奶香 / 果味 / 茶感 / 咖啡感 / 微醺 / 低酒精 / 无酒精

输出页面不是“某某鸡尾酒”，而可以是“便利店清爽微醺方案”。

方案详情包括：

- 购买清单
- 预计价格
- 调制步骤
- 风味描述
- 替换方案
- 适合场景
- 翻车提醒
- 相似方案

后续实现中，便利店模式被改成套餐卡，突出预算、杯数、成功率和购物清单感。

### 家里有什么页

这是最强的工具型功能。

用户勾选手上已有材料：

- 基酒：金酒、伏特加、威士忌、朗姆、百利甜、梅酒
- 饮料：可乐、雪碧、气泡水、橙汁、茶、咖啡、牛奶
- 辅助：冰块、柠檬、糖、盐、水果

输出结果分层：

- 现在直接能做
- 只差 1 样材料
- 便利店补一下就能做
- 更好喝的升级版

### 新手推荐页

服务“不知道搜什么”的用户，不做百科，做选择题。

推荐流程：

1. 你能接受酒味吗？
2. 你喜欢什么味道？
3. 你现在方便怎么买？

输出新手推荐酒单。

### 标准详情页

所有结果最终落到标准详情页。

结构包括：

- 标题区：如“金汤力 Gin Tonic”
- 标签：清爽 / 微苦 / 气泡感 / 新手友好 / 夏天
- 一句话推荐理由
- 风味雷达或 5 维评分：甜度、酸度、酒感、清爽度、难度
- 材料清单：标准版、便利店替代版、家里简化版
- 调制步骤：控制在 3-5 步
- 适合场景
- 替代方案
- 相似推荐：更甜、更清爽、更适合新手、更便宜、更便利店

## 4. 三层产品架构

后续讨论把产品定成三层体验：

### Layer1：品牌情绪启动层

职责：让我想喝。

用于建立品牌世界观和消费欲，不承担高频操作。允许夏夜便利店、大饮品视觉、冷光、冰块、青柠、气泡、冷凝水和情绪文案，但只保留一个明确进入动作。

### Layer2：真正首页

职责：让我快速找到喝法。

用户已经进入“我要找方案”的状态，所以必须出现搜索框、高频搜索词、三个主入口和工具感信息。

三大主入口：

- 家里有什么
- 便利店调酒
- 按基酒找

### Layer3：搜索结果 / 配方决策层

职责：让我快速决定怎么喝。

结果页第一屏必须给出最推荐的一杯、大图、风味标签、一句话推荐理由和查看配方动作。扩展推荐只做辅助，不能抢主推荐。

## 5. 视觉方向

统一视觉基调：

- 奶油纸感
- 轻颗粒
- 透明空气感
- 柔和边缘
- 低饱和色
- 少量青绿色品牌色

项目中已经有配套文档：

- `DESIGN_SYSTEM.md`：页面结构、按钮、标签、插画、节奏、动效。
- `FLAVOR_VISUAL_MAPPING.md`：不同酒类和场景的视觉语言。
- `VISUAL_ASSET_SPEC.md`：视觉资产规格。
- `utils/visualSystem.js`：代码侧主题配置。

不同酒类视觉语言：

- 金酒：青柠、冰感、气泡、透明、青绿色冷光。
- 威士忌：琥珀、木质、夜晚、冰球、暖光。
- 百利甜：奶香、咖啡、巧克力、甜品感。
- 伏特加：透明、冷蓝、冰块、柠檬。
- 朗姆：热带水果、薄荷、橙黄、酸甜。
- 便利店调酒：冰柜、夜晚便利店、小瓶饮料、冷光、即时满足。

重要限制：

不要全站都是大海报。品牌感来自统一的色彩、材质、光感、字体层级和饮品语言，而不是每页重新做一个 KV。

## 8. 调试与风险记录

曾重点排查过“搜索金汤力后结果页显示不正确”的问题。

问题风险来自云端数据缺失或别名映射命中 ID 后找不到对应配方，导致搜索逻辑降级到模糊匹配，可能命中其他金酒配方。

已做的修复方向：

- `utils/recommend.js` 增加 ID 命中失败后的中文名精确匹配降级。
- `pages/results/results.js` 增加别名 ID 失败后的第三重兜底。

## 10. 后续继续开发建议

优先级建议：

1. 先用微信开发者工具导入当前目录，确认首页、搜索、详情、便利店调酒、家里有什么都能打开。
2. 检查 `cloud-data/` 是否需要重新导入微信云数据库。
3. 如果继续做视觉，优先统一 Layer2 和 Layer3，不要再新增大海报式页面。
4. 如果继续做产品功能，优先优化搜索结果决策、便利店套餐、家里有什么的缺料分层。
5. 如果继续做内容，优先补全常见搜索词对应的配方和材料长尾。
